# muu-cardcraft Completion Plan

## 目的

`muu-cardcraft` を、ポートフォリオに掲載できる完成作品として整理する。

現状は名刺編集機能の多くが完成に近く、課題は主に「表裏プレビューから購入までの導線」をどこまで完成扱いにするかにある。

この資料では、フルEC連携を完成条件に含めず、まずは「入稿前プレビュー付き名刺デザインエディター」として完成させる方針を定義する。

## 現在地

`muu-cardcraft` には、すでに次の導線がある。

-   名刺デザイン編集
-   表面 / 裏面のプレビュー
-   画像アップロード
-   テキスト編集
-   レイヤー操作
-   PNG / JPEG 書き出し
-   order preview
-   Supabase への design 保存
-   Next.js の `/api/orders` から WordPress bridge を呼ぶ入口

つまり、エディターとしての体験はかなり進んでいる。

一方で、WordPress / WooCommerce 側のカート追加、checkout、order meta への `design_id` 引き継ぎまでは未完成。

## 完成ライン

第一リリースでは、購入完了までを完成条件にしない。

完成ラインは次の範囲に置く。

``` text
編集
  -> 表裏プレビュー
  -> 注文前データ確認
  -> デザイン保存
  -> 入稿/相談用データの準備
```

この段階では、`muu-cardcraft` を「購入できるEC」ではなく、「注文前のデザイン確認と入稿準備ができるエディター」として扱う。

## 完成版として含めるもの

### 編集機能

-   テキスト追加 / 編集
-   フォント変更
-   色変更
-   画像追加
-   画像移動 / リサイズ
-   レイヤー順変更
-   表面 / 裏面の切り替え
-   ガイド表示

### プレビュー

-   編集中プレビュー
-   表面 / 裏面の注文前プレビュー
-   ガイド表示とは別の仕上がり確認
-   スマホでも確認しやすい表示

### 書き出し

-   PNG 書き出し
-   JPEG 書き出し
-   必要に応じて表裏別の書き出し

### 保存

-   order preview へ進む時点でデザインを保存する
-   画像は長期参照できる形へ寄せる
-   Supabase 側に design / draft を残す

### 注文準備

第一リリースでは「購入」ではなく「注文準備」として扱う。

-   表面と裏面を確認する
-   保存済み design の参照IDを持つ
-   必要なら相談や手動対応へ渡せる
-   WordPress bridge が未設定でもアプリが破綻しない

## 第一リリースで外すもの

次の項目は、第一リリースの完成条件から外す。

-   WooCommerce カートへの自動追加
-   checkout への自動遷移
-   決済処理
-   WordPress 注文詳細への完全連携
-   cart item meta / order item meta への `design_id` 引き継ぎ
-   管理者向け注文管理画面

これらは重要だが、ECとしての責任範囲が大きい。

ポートフォリオでは後続拡張として説明する。

## 購入導線の扱い

現状の「カートに追加」は、完成版では文言を見直す。

まだWooCommerce連携が完成していない場合、ユーザーに購入完了を期待させない表現にする。

候補:

-   `注文内容を確認`
-   `入稿データを作成`
-   `デザインを保存`
-   `相談用データを作成`
-   `この内容で相談する`

第一リリースでは、`カートに追加` よりも `入稿データを作成` が自然。

## 推奨フロー

### 通常フロー

``` text
1. 名刺を編集する
2. 表面 / 裏面を確認する
3. 入稿データを作成する
4. Supabase に design を保存する
5. 完了画面で designId / draftCode を表示する
6. 画像を書き出す、または相談導線へ進む
```

### WordPress bridge が設定済みの場合

``` text
1. 名刺を編集する
2. 表面 / 裏面を確認する
3. Supabase に design を保存する
4. Next.js /api/orders から WordPress bridge を呼ぶ
5. WordPress 側に card_design 投稿を作る
6. checkoutUrl が返れば外部導線として表示する
```

ただし、この場合でも WooCommerce 注文完了までは第一リリースの完成条件に含めない。

## UI方針

### order preview

order preview は「購入直前」ではなく「入稿前確認」として扱う。

表示する内容:

-   オモテ面
-   ウラ面
-   注意書き
-   保存 / 入稿データ作成ボタン

注意書きの例:

``` text
この画面では仕上がりイメージを確認できます。
文字や画像の位置は、編集画面のガイドもあわせて確認してください。
```

### 完了画面

保存後は、注文完了ではなく「デザイン保存完了」として見せる。

表示する内容:

-   designId
-   draftCode
-   書き出しボタン
-   編集に戻る
-   相談する / 問い合わせる

WordPress bridge が成功した場合だけ、追加情報として WordPress post ID や checkout URL を表示する。

## Supabase / WordPress の役割

### Supabase

第一リリースでも使う。

-   design の正本を保存する
-   draftCode を発行または保持する
-   画像の長期参照を確保する
-   再編集や手動対応の起点にする

### WordPress

第一リリースでは任意連携として扱う。

-   bridge が設定されていれば `card_design` 投稿を作る
-   bridge が未設定でもエディターの完成度に影響させない
-   WooCommerce への完全接続は後続タスクにする

### WooCommerce

後続フェーズで扱う。

-   cart item meta に `design_id` を残す
-   checkout 完了時に order item meta へ引き継ぐ
-   注文詳細から Supabase の design を辿れるようにする

## ポートフォリオでの説明

ポートフォリオでは、次のように表現する。

``` text
muu-cardcraft は、ブラウザ上で名刺デザインを作成し、
表裏プレビューと入稿用データ作成まで行えるWebエディターです。
Supabase にデザインを保存し、将来的な WordPress / WooCommerce 連携を見据えた構成にしています。
```

購入機能を前面に出すより、編集体験と設計の拡張性を伝える。

## 実装順

### Step 1. 文言を購入から入稿準備へ寄せる

`カートに追加` などの文言を見直す。

第一リリースでは、購入完了を想起させる言葉を避ける。

### Step 2. WordPress bridge 未設定時の表示を整える

`wordpress_order_bridge_not_configured` が返った場合も、失敗ではなく「デザイン保存は完了、外部連携は未設定」と分かるようにする。

### Step 3. 保存完了画面を用意する

order preview から進んだ後、designId / draftCode を確認できる完了表示を作る。

### Step 4. 書き出し導線を明確にする

保存後に PNG / JPEG を書き出せる導線を用意する。

表面 / 裏面のどちらを書き出すかが分かるようにする。

### Step 5. README / docs / スクリーンショットを整える

ポートフォリオ掲載に向けて、完成範囲を明記する。

## 注意点

-   第一リリースではECとして完成させようとしない
-   購入できるように見せすぎない
-   Supabase保存までは完成範囲に含める
-   WordPress / WooCommerce は後続拡張として扱う
-   失敗表示は「購入失敗」ではなく「外部連携未設定」として整理する
-   完成作品として見せるときは、編集体験と入稿準備に焦点を当てる

## ゴール

`muu-cardcraft` を、購入フロー未完成のプロトタイプではなく、入稿前プレビュー付きの名刺デザインエディターとして完成させる。

そのうえで、Supabase / WordPress / WooCommerce 連携は、次フェーズの拡張として自然に続けられる状態にする。
